Collivind — Architecture Approaches

Three approaches for building an AI coding session memory layer with graph-based knowledge

A — Monolith MCP Server

SIMPLEST

Single Python process serves MCP tools, manages Qdrant/Neo4j connections, runs embeddings. Everything in one server.

Pros

  • Fastest to build and ship
  • Single process to debug
  • Simple Docker Compose

Cons

  • Hard to swap backends later
  • Enterprise fork = major rewrite
  • Tight coupling everywhere
PythonMCP SDKQdrantNeo4jHuggingFace

B — Layered Core + MCP Shell

BALANCED

Core library with pluggable storage interfaces. MCP server is a thin shell that calls the core. Enterprise swaps backends via config.

Pros

  • Pluggable backends from day 1
  • Enterprise = config change
  • Core is testable in isolation
  • Clean separation of concerns

Cons

  • More upfront design work
  • Interface abstraction overhead
PythonMCP SDKQdrantNeo4jHuggingFaceInterfaces

C — Microservices

MOST SCALABLE

Separate services for vector search, graph, embeddings, and MCP gateway. Each independently deployable.

Pros

  • Maximum flexibility
  • Each service scales independently
  • Cloud-native from the start

Cons

  • Way over-engineered for v1
  • 5+ Docker containers locally
  • Complex networking and debugging
  • Slow to ship
